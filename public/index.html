releases.io

Maintaining your project's release notes in static HTML files for software update frameworks can be a chore. If your project is hosted on GitHub you may have adopted GitHub Releases, in which case you're already halfway to solving the problem. Wouldn't it be great if you could show those same release notes in your updater, without the unnecessary GitHub interface?

Usage

To use releases.io, construct a URL with the following format: 

https://releases.io/:user/:repo/:tag

Specify the following parameters in the URL:

Parameter Description
:user The GitHub username associated with the project.
:repo The GitHub repository name for the project.
:tag The git tag associated with the release notes (e.g. v0.3.0 or 4.1.0).

If a tag has no release notes associated with it (or the repository name or username is misspelled or non-existent) a 404 response code will be returned.

If you want to display only the most recent release notes for your project, rather than those for a specific version, just omit the tag parameter from the URL:

https://releases.io/:user/:repo

Once you have a working link, simply add it to your software updater. For example, if you're using Sparkle, add the URL to your appcast file:

...
<sparkle:releaseNotesLink>
  https://releases.io/:user/:repo/:tag
</sparkle:releaseNotesLink>
...

Both HTTP and HTTPS protocols are supported for all releases.io links. If you'd like to have more control over how the notes look, refer to the Custom Styles section.

GitHub Compatibility

In addition to the short-form URL described above, a longer variation is available for compatibility with GitHub's naming scheme. This makes it incredibly easy to turn a GitHub Releases link into a valid releases.io link by replacing just the top-level domain name:

https://github.com/yui/pure/releases/tag/v0.3.0
Becomes: 
https://releases.io/yui/pure/releases/tag/v0.3.0

Custom Styles

An external stylesheet can be used to control the appearance of your release notes by specifying a css URL parameter: 

https://releases.io/:user/:repo/:tag?css=http://yourhost.com/style.css

If you're only concerned with adjusting the base font style, you can use one of two basic pre-defined stylesheets:

  • css=sans - for standard sans serif fonts
  • css=serif - for standard serif fonts

Using an external stylesheet adds a small overhead to the page load time, which you may want to take into consideration.

Advanced

The following optional URL parameters are available:

Parameter Description
ugly= When true, no formatting or indentation is applied to the HTML code. This is the default value. Specify false for better-formatted HTML code, at the cost of slower response times.

Caveats

Only published release notes work with releases.io — draft release notes are only accessible once published.

The 'Why?'

The motivation behind releases.io was a combination of several aims:

  • To promote and encourage the use of GitHub Releases for open-source projects, because hosting your release notes at the same location that you host your code just makes sense.
  • To supplant the need to maintain many static HTML files for software update frameworks — when no particular pizzaz is required — and instead use release notes generated directly from those hosted on GitHub.
  • To provide a quick and easy way of sharing release notes with other people, without the full-blown GitHub web interface, putting the focus squarely on the content.
  • For fame and profit — which may or may not be true.
