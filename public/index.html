releases.io

Maintaining release notes as static HTML files for software update frameworks such as Sparkle can be a chore, but it doesn't have to be that way. If you're hosting your code on GitHub, you may already be using GitHub Releases for your release notes, in which case you're already halfway to solving the problem. Wouldn't it be great if you could display those very same release notes in your updater, without the unnecessary GitHub web interface? Enter releases.io.

You can retrieve a plain HTML copy of your release notes, without the GitHub interface, using just a simple URL, for displaying in your application's software updater or using elsewhere.

URLs are formatted like this:

https://releases.io/:user/:repo/:tag

Both HTTP and HTTPS protocols are supported. The parameters that make up the URL are explained below:

Parameter Description
:user the Github username for the project maintainer
:repo the Github repository name for the project
:tag the git tag associated with the release notes (e.g. v0.3.0 or 4.1.0)

The resulting HTML page is rendered directly from the raw GitHub Flavoured Markdown release notes hosted on GitHub. If a tag has no release notes associated with it (or the repository name or username is mispelled or non-existent) a 404 response code will be returned.

If you'd like a link that always displays the most recent release notes for a project, just omit the tag parameter from the URL:

https://releases.io/:user/:repo

In addition to the short-form URL described so far, a longer variation is available that is directly compatible with GitHub's naming scheme. This makes it incredibly easy to turn a GitHub Releases link into a valid releases.io link by replacing the top-level domain name.

For example, the following GitHub link:

https://github.com/yui/pure/releases/tag/v0.3.0
Can be modified to: 
https://releases.io/yui/pure/releases/tag/v0.3.0

Note: The words 'releases' and 'tag' are ommited from the short-form URL, and only included in the API for GitHub link compatibility.

Warning: using this service in production-ready systems or code is not recommended at present due to its use of the developer preview of the GitHub Releases API. This message will be removed once the API has been finalised.

Why?

The motivation to build releases.io was a combination of several aims:

  • To promote and encourage the use of GitHub Releases for open-source projects, because hosting your release notes at the same location that you host your code just makes sense.
  • To supplant the need to maintain many static HTML files for software update frameworks — when no particular pizzaz is required — and instead use release notes generated directly from those stored on GitHub.
  • To provide a quick and easy way of sharing release notes with other people, without the full-blown GitHub web interface, putting the focus squarely on the content.
  • For fame and profit — which may or may not be true.

Custom Styles

By default, no styling is applied to the pages generated by this service. If you would like to retain an element of control over the look of your pages you can specify an external stylesheet in a css URL parameter: 

https://releases.io/yui/pure/releases/tag/v0.3.0?css=http://host.com/style.css

If you're only concerned with the base font style, you can use one of the following pre-defined stylesheets:

  • css=sans - uses standard sans serif fonts
  • css=serif - uses standard serif fonts

Using an external stylesheet may add a small overhead to the page load time.
